# UI 디자인 브리프 — LLM Dev Control Tower (v1)

- 목적: 이 문서를 **Claude(또는 frontend-design 도구)에 그대로 전달**해 v1 UI를 설계받기 위한 브리프.
- 기준 설계: [`2026-06-28-llm-dev-control-tower-design.md`](2026-06-28-llm-dev-control-tower-design.md) (§8 UI). 충돌 시 그 spec이 우선.
- 범위: **시각/레이아웃 설계만** (화면·컴포넌트·상태·인터랙션·비주얼 방향). 구현 코드는 별도.

---

## 0. 붙여넣기용 프롬프트 (Claude design에 이걸 먼저 주면 됨)

> 나는 **"LLM Dev Control Tower"** 라는 단일 사용자 로컬 개발자 도구의 UI를 설계하려 한다. 이건 LLM(Claude/Codex)이 큰 코딩 프로젝트를 **작고·보이고·리뷰 가능한 step**으로 진행하게 하고, 진행하면서 **목표→티켓→step→코드영역→테스트→결정**을 잇는 **살아있는 지도**를 짓는 시스템이다. 핵심 가치는 *자동화*가 아니라 **가시성·통제·외과적 디버깅**이다(블랙박스 탈출). 아래 브리프의 화면 4개(① 프로젝트 지도 ② 티켓 레인 ③ 리뷰 페인 ④ 버그추적)를, 명시된 상태/데이터/인터랙션/안티패턴을 지켜 **와이어프레임 + 컴포넌트 상태 + 비주얼 방향**으로 설계해 달라. 스택은 React + Vite + 그래프 뷰 라이브러리(React Flow류)라 그 안에서 구현 가능해야 한다. 미니멀하지만 템플릿처럼 보이지 않는, 정밀한 *개발자 콘솔* 톤으로.

---

## 1. 이 제품이 뭔가 (한 문단)

거친 목표("Todo 앱: 할일 CRUD")를 주면, 시스템이 LLM을 작은 step으로 일하게 하고 step마다 **정지**해 사람(Kay)이 리뷰한다. 일하는 동안 git diff에서 **기능↔코드 지도**를 자동으로 짓는다. 뭔가 깨지면 지도가 "어디를 볼지" 답한다. **단일 사용자, 로컬, 데스크톱.** 사용자는 LLM로 큰 프로젝트를 만들다가 *결과 + 텍스트 덩어리만 받아 어디서 QA할지/버그가 어딘지 모르는* 고통을 겪은 개발자다.

## 2. 디자인 북극성 (이 느낌을 만들어라)

- **지도가 홈.** 칸반/리스트가 아니라 **그래프(지도)** 가 1차 화면. 프로젝트를 지도로 항해한다.
- **살아있고 정직한 상태.** 지금 무엇이 도는 중인지/멈췄는지가 실시간 반영(수동 새로고침 없음). "지금 step 3 실행 중", "리뷰 대기 중"이 한눈에.
- **리뷰가 주 작업면.** 사용자가 시간을 가장 많이 쓰는 곳은 *step 리뷰 게이트*. 여기를 가장 공들여.
- **배관 0.** 내부 동작(워커/큐 tick 버튼, 파일 절대경로, API URL, 액터명 같은 디버그 표면)을 **절대 노출하지 않는다.**
- **정밀한 개발자 콘솔 톤.** 미니멀·고밀도이되 템플릿 같지 않게. 차분하지만 살아있음. 코드/식별자는 모노스페이스. 다크 우선(라이트도 고려).

## 3. 절대 하지 말 것 (전작 v2의 불편함 = 안티패턴)

- ❌ `Worker tick` / `Watcher tick` 같은 **내부 배관 버튼**.
- ❌ 수동 `Refresh` 의존(상태가 stale하게 보임).
- ❌ 한 티켓을 보려고 **세로로 끝없이 쌓인 폼 더미**(드로어 스크롤 지옥).
- ❌ wiki 파일경로·API 주소·`actor=...` 같은 **운영자 디버그 표면 노출**.
- ❌ 드래그=실행인데 떨궈도 **아무 시각적 반응이 없는** 무피드백.

## 4. 핵심 사용자 흐름 (이걸 화면이 지원해야)

**A. 한 티켓 수직 루프 (주 흐름)**
```
목표 입력 → [PLAN] 에이전트가 step 목록 제안 → 사용자가 분해 편집·승인
→ [각 STEP] 시스템 자동 실행 → 끝나면 정지 → 사용자 리뷰(✅승인 / ✏️수정요청 / ⏸인수)
→ 다음 step … 반복 → 티켓 완료
```
**B. 버그 추적 제스처 (킬러 기능)**
```
파일/심볼/UI요소를 클릭하거나 검색 → 지도가 "그걸 만든 step → 티켓 → 결정" 경로를 하이라이트
→ 해당 commit/diff로 점프
```

## 5. 설계할 화면 (메인)

### ① 프로젝트 지도 (홈 · zoom-out)
- **그래프 캔버스.** 중앙/상단에 `Objective`, 가지로 `Ticket`들. 각 티켓에 **step 진행도(N/M)** 와 **상태색**.
- `CodeRegion`(파일/심볼) 레이어는 **토글**(켜면 티켓↔코드 연결선이 보임).
- 한눈에 "지금 어디·무엇이 도는 중·무엇이 막힘".
- 티켓 클릭 → ② 티켓 레인으로 zoom-in.
- 상단에 **목표/티켓 추가** 진입점, **버그추적 검색**(④) 입력.

### ② 티켓 레인 (zoom-in · 작업 뷰)
- 한 티켓의 **step 타임라인**(수직 or 수평). 각 step = 카드.
- 정지(리뷰 대기)한 step이 **전면 중앙**으로 강조.
- 각 step 카드: 번호·의도(요약)·상태·touched 코드영역 미리보기.
- step 클릭/리뷰대기 → ③ 리뷰 페인.

### ③ 리뷰 페인 (게이트 · **가장 중요**)
- **2분할.** 좌: **diff**(문법 하이라이트). 우: 이 step이 만든 **지도 조각**(새 노드/엣지) + 에이전트 **결정/왜(Decision)** + **acceptance 체크**.
- 하단 고정 **3액션**: `✅ 승인` / `✏️ 수정요청(코멘트)` / `⏸ 내가 인수`.
- ✏️ 누르면 코멘트 입력 → 그 step 재실행. ⏸는 "내가 직접 편집" 모드.
- 옛 드로어의 세로 폼 더미 → **한 화면 결정 뷰**로 대체.

### ④ 버그 추적 (지도 위 오버레이/모드)
- 검색창 또는 노드 클릭으로 진입.
- 지도가 **소유 경로(CodeRegion→Step→Ticket→Decision)** 를 하이라이트, 나머지는 디밍.
- 해당 commit/diff로 점프하는 액션.

### (보조) 플랜 승인 뷰
- PLAN 직후 `interrupt` 지점: 제안된 step 목록을 **편집(추가/삭제/병합/순서)** 후 승인. 가벼운 모달/패널이면 충분.

### (보조) 진입/목표 입력
- 프로젝트 목록 + "새 목표" 입력(거친 자연어 한 줄~문단).

## 6. 노드/상태 비주얼 (디자이너가 정의할 것)

**노드 6종** (각각 구별되는 시각 토큰 필요):
`Objective` · `Ticket` · `Step` · `CodeRegion`(파일/심볼) · `Test` · `Decision`

**엣지**: `has` · `subdivides`(재귀) · `touches`(step→코드, 진실원천) · `tested_by` · `decided` · `produced(commit)`

**상태색 팔레트** (티켓/step 공통, 한눈에 구분되게):
`planning` · `executing`(●live 느낌) · `⏸ awaiting-review`(주목) · `done` · `blocked`(실패)

## 7. 화면이 바인딩하는 데이터 (뭐가 떠야 하는지)

- 지도: 노드 목록 + 엣지 목록 + 각 노드 상태.
- 티켓 레인: 티켓의 step[] (상태·의도·touched CodeRegion).
- 리뷰 페인: step의 **diff(텍스트)** + 이번 step이 만든 노드/엣지 + Decision(why) + acceptance + 3액션.
- 버그추적: 임의 노드에서의 역방향 경로.
- 실시간성: 티켓 라이프사이클 상태(실행/정지)를 **구독**해 자동 반영(폴링/스트림은 구현 디테일, 디자인은 "항상 최신"을 전제).

## 8. 인터랙션 원칙

- **2고도 항해**: zoom-out(프로젝트 지도) ↔ zoom-in(티켓 레인) 매끄럽게.
- **상태는 늘 살아있음**: 진행/정지 전이가 즉시 반영, 사용자가 새로고침 안 함.
- **키보드 친화**(승인/다음/수정요청 단축키) 고려 — 개발자 도구다.
- **반응형**: 데스크톱 우선(1080p~4K 유동), 멀티패널이 넓은 화면에서 동시에(지도+레인+리뷰).

## 9. 기술 제약 (설계가 구현 가능해야)

- React + Vite + **그래프 뷰 라이브러리(React Flow류)** 안에서 구현 가능한 지도.
- 과한 모션/3D 지양 — 성능·명료성 우선. 모션은 *상태 전이를 읽히게 하는* 용도로만.

## 10. 이 디자인 패스에서 받고 싶은 산출물

1. 화면 4개(+보조 2개)의 **와이어프레임/레이아웃**.
2. **노드 6종·엣지·상태색**의 비주얼 토큰(범례).
3. **리뷰 페인** 상세(2분할 + 3액션 + 수정/인수 상태).
4. **네비게이션 모델**(zoom-out↔zoom-in, 버그추적 진입).
5. **비주얼 방향 1~2안**(컬러/타이포/밀도) — 개발자 콘솔 톤.
6. (선택) 빈 상태/로딩/에러/blocked 등 **엣지 상태**.

## 11. 디자이너에게 남기는 열린 질문

- 지도 레이아웃: 자동 배치(force/layered) vs 사용자 고정 배치 — 무엇이 "버그→어디" 항해에 더 읽히나?
- diff 표시: 인라인 vs 좌우 split — 리뷰 페인 2분할과 어떻게 공존?
- 큰 프로젝트(노드 수백)에서 지도의 **밀도 관리**(클러스터링/포커스+컨텍스트)?
- `Step` 정지 강조를 지도(①)에서도 보여줄지, 레인(②)에서만 보여줄지?
