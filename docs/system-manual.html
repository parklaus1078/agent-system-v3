Agent System v3 — 살아있는 조종석(Living Cockpit)

거친 목표를 작고·보이고·리뷰 가능한 step으로 분해해 실행하는 LLM 개발 관제탑. 인과 지도(causal map) 위에서 계획→실행→리뷰가 흐르고, 사람은 자율도 다이얼과 대화 채널(steer)로 개입한다. 이 문서는 UI/서버의 모든 기능을 영역별로 나열하고, 각 기능에 쓰인 핵심 기술 스택과 소프트웨어 개념을 정리한 서비스 매뉴얼이다.

프론트엔드(UI) 백엔드 API 백엔드 내부 tech 기술 concept 개념

1. 시스템 개요

두 개의 직교 축이 UI 전체를 구동한다: 노드 종류(kind) → 모양/아이콘, 상태(status) → 색. 백엔드는 프로젝트별 인과 그래프(Node/Edge)를 저장하고, 티켓마다 LangGraph 상태기계로 계획→실행→리뷰를 HITL(Human-in-the-loop) 게이트와 함께 돌린다.

도메인 모델 — 인과 그래프

6종 노드 · 6종 엣지

노드: objective(목표) · ticket(일감) · step(단위작업) · code_region(코드 영역) · test(테스트) · decision(결정/제약). 엣지: has · subdivides · touches · tested_by · decided · produced. 상태는 planning / executing / awaiting_review / done / blocked.

concept directed graphdomain modeling two-axis visual encoding

핵심 루프 — 계획 · 실행 · 리뷰

HITL lifecycle

목표 입력 → 티켓/플랜 제안 → 사람이 승인 → step 단위 실행(코드 diff 생성) → 리뷰 게이트(승인/수정요청/인수). 자율도에 따라 게이트가 매 step / 실패·최종만 / 무정지로 조절된다.

concept state machinehuman-in-the-loop interrupt / resumeone step = one commit

2. 아키텍처 & 데이터 흐름

3계층 + 어댑터 seam. 프론트는 ApiClient 인터페이스에만 의존하고, 실제 백엔드(HttpApiClient)와 브라우저 목업(MockApiClient)을 런타임에 교체한다. 모든 쓰기는 subscribe → reload로 라이브 반영된다.

steer 한 번의 흐름: 채널 입력(자유 NL) → POST /steer → intent-router가 7개 고정 op 중 하나로 분류 (redirect·constrain·reprioritize·scope·answer·control·clarify) → 그래프/자율도/체크포인트에 결정적으로 반영 → 채널에 결과 메시지 + 지도 하이라이트. 사람의 말이 항상 되돌릴 수 있는 그래프 조작으로만 반영된다.
실행 모드 & 결정성: ASV3_AGENT_MODE=simulated에서 planner/executor/intent-router가 결정적으로 동작(테스트·데모). anthropic-api/claude-cli 등으로 지점별 교체 가능. 비동기 워커는 governance를 요청 스레드에서 해석 + 스폰 전 commit으로 공유 연결 레이스를 회피한다.

4. 최근 추가 기능 2026-07-01

v1 이후 추가된 3개 기능. 각 기능은 아래 상세 섹션(FE/BE)에도 반영되어 있다.

① 티켓 id = {slug}-{번호} 자동 증가

store.next_ticket_id · domain/graph.nextTicketId

모든 티켓 생성 경로(프로젝트 승인 · steer scope · UI 새 목표)가 일관된 {slug}-{n} 형식으로 id를 부여한다. n = 기존 최대 티켓 번호 + 1(auto-increment, 충돌 안전). 예전 UI 경로의 불투명한 t-{timestamp}를 없앴고, 정렬 정규식은 -t?(\d+)$로 레거시 -t{n}도 호환. 프론트 nextTicketId가 백엔드 next_ticket_id를 미러한다.
